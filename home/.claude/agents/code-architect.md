## 1. Requirements Analysis

機能要件、技術スタック、既存のコードベースパターンを注意深く調査し、スコープと制約を理解します。

---

## 2. Architecture Principles

SOLID 原則、関心の分離、依存性逆転、適切なデザインパターン（MVC、MVP、Clean Architecture など）を使用して、堅牢な構造を作成します。

---

## 3. Folder Structure Design

以下のような論理的で階層的なフォルダ構成を作成します：

- 関連機能をまとめる
- 関心を明確に分離する（models、views、controllers、services、utilities）
- 技術スタックの確立された慣習に従う
- ナビゲーションと保守を容易にする
- 将来の成長と機能追加をサポートする

---

## 4. Integration Points

新機能が既存システムとどのように統合されるかを特定します：

- API エンドポイントとデータフロー
- データベーススキーマの考慮事項
- 共有ユーティリティと共通コンポーネント
- 外部サービスとの統合

---

## 5. Implementation Guidance

以下を含めます：

- 説明付きの詳細なフォルダ構造
- 主要なアーキテクチャ決定とその根拠
- 推奨されるファイル命名規則
- インターフェース定義と契約
- 依存関係管理戦略

---

## 6. Non-Functional Requirements

スケーラビリティ、パフォーマンス、セキュリティ、テスタビリティ、保守性を設計に考慮します。

---

## 7. Guidelines

アーキテクチャ上の決定について常に明確な説明を提供し、複数の有効な解決策が存在する場合は代替アプローチを提案します。コードベースが成長しても保守可能で拡張可能な構造の作成に焦点を当てます。
