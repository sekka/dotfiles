# 提案資料作成ガイドライン

このファイルは、効果的な提案資料を作成するための詳細なガイドラインを提供します。

---

## リード文の書き方

### 基本原則

リード文はスライドの核。**ここで8割決まる。**

1. **スライドは必ずリード文から書く**
2. **リード文だけで内容がわかるくらい具体的に書く**
3. **コンテンツを変えたら必ずリード文を見返す**

### 良いリード文の例

```
❌ 悪い例:
「下記に示す通り、分析結果を整理しました」

✅ 良い例:
「顧客離脱の主因は初回利用時のUX不良であり、オンボーディング改善で30%の離脱防止が可能です」
```

### リード文チェックリスト

- [ ] このスライドで最も伝えたいことが1文で表現されているか
- [ ] 読み手がリード文だけで内容を理解できるか
- [ ] 「以下の通り」「下記に示す」的な逃げの表現を使っていないか
- [ ] 具体的な数字や結論が含まれているか

## 構成設計のルール

### スライドの基本構造

```
┌─────────────────────────────────┐
│ 見出し（結論を述べる）          │
├─────────────────────────────────┤
│ リード文（詳細な結論・示唆）    │
├─────────────────────────────────┤
│                                 │
│ コンテンツ（根拠・詳細）        │
│                                 │
└─────────────────────────────────┘
```

### 1スライド1メッセージ

| ルール | 理由 |
|--------|------|
| 見出しで結論を述べる | 読み手は見出しを最初に見る |
| 本文は補足情報のみ | 情報過多を避ける |
| 箇条書きは3〜5項目まで | 認知負荷を下げる |

## 論理構成パターン

### 1. PREP法（結論先行型）

```
P - Point（結論）: 「A案を推奨します」
R - Reason（理由）: 「なぜなら〜だからです」
E - Example（具体例）: 「実際に〜という事例があります」
P - Point（結論）: 「したがってA案を推奨します」
```

**適用場面**: 経営層への報告、意思決定を求める提案

### 2. 問題解決型

```
現状 → 課題 → 原因 → 解決策 → 効果
```

**適用場面**: 改善提案、コンサルティング提案

### 3. 比較検討型

```
選択肢の提示 → 評価軸の定義 → 比較分析 → 推奨案
```

**適用場面**: ツール選定、ベンダー選定

## デザインルール詳細

### 色使い

| ルール | 具体例 | 理由 |
|--------|--------|------|
| 黒は使わない | `#333333` を使用 | 視認性と柔らかさのバランス |
| 強調色は1-2色 | 青と赤のみ | 色が多いと散漫になる |
| ○×は全部に色つけない | ○は無色、×のみ赤 | 主張点を明確に |
| グレーアウトは数値消す | グレー部分の%は削除 | ノイズを減らす |

### カラーパレット推奨

```
メインカラー: #2563EB (青)
アクセント: #DC2626 (赤)
テキスト: #333333 (ダークグレー)
背景: #FFFFFF (白)
サブ背景: #F3F4F6 (ライトグレー)
罫線: #E5E7EB (グレー)
```

### レイアウト

- **ガイドからはみ出さない**: 余白を統一する
- **オブジェクトは配置機能で揃える**: 目測で揃えない
- **余白は最低20px**: 窮屈な印象を避ける

### 表・グラフ

| ルール | 理由 |
|--------|------|
| 積み上げ棒グラフは横にする | 縦だとラベルが読みにくい |
| 表の列名の線を入れない | ヘッダーは背景色で区別 |
| 罫線はグレーにする | 黒は重い印象 |
| 数値の合計を確認 | 部分と合計の整合性 |

### フォント

| 要素 | サイズ | ウェイト |
|------|--------|----------|
| タイトル | 32pt | Bold |
| 見出し | 24pt | Bold |
| リード文 | 18pt | Regular |
| 本文 | 16pt | Regular |
| 注釈 | 14pt | Regular |
| **最小** | **12pt** | - |

**12pt以下は絶対に使わない**

## 文章表現ルール

### 避けるべき表現

| NG表現 | 理由 | OK表現 |
|--------|------|--------|
| 「〜について」 | 冗長 | 直接述べる |
| 「〜に関して」 | 冗長 | 直接述べる |
| 「以下の通り」 | 逃げ | 具体的な結論 |
| 「様々な」 | 曖昧 | 「3つの」など具体化 |
| 「等」 | 曖昧 | 列挙するか省略 |

### 「ただし」と「なお」の使い分け

- **ただし**: 前の文節に対して釘を刺す時

  ```
  本施策は効果的です。ただし、初期投資が必要です。
  ```

- **なお**: 新規の話題や補足情報を付け足す時

  ```
  本施策は効果的です。なお、詳細は別紙をご参照ください。
  ```

## 資料作成プロセス

### フェーズ1: 骨子作成

1. 目的を1文で定義
2. 聴衆を特定
3. 核心メッセージを決定
4. 大まかな構成（章立て）を決定

### フェーズ2: スライド作成

1. 各スライドのリード文を先に書く
2. リード文を支える根拠を追加
3. ビジュアル（図・グラフ）を作成
4. スピーカーノートを追加

### フェーズ3: レビュー・修正

1. リード文だけを通して読む（ストーリーが通るか）
2. デザインチェックリストで確認
3. 第三者にレビュー依頼
4. スライドショーで最終確認

## コミュニケーションルール

### 確認の仕方

| NG | OK |
|----|-----|
| 「わかりました」 | 「〜という理解で合っていますか？」 |
| 「どうしたらいいですか？」 | 「AとBの案がありますが、どちらが適切でしょうか？」 |

### フィードバックの受け方

- 一度言われたことは繰り返さない
- 指摘を受けたら修正案もセットで報告
- 「レビュー」ではなく「すり合わせ」「認識合わせ」を使う

### 手離れの良い資料

- 仕掛かり中のスライドには状態を明記

  ```
  【作成中】データ待ち
  【要確認】A案/B案どちらにするか
  ```

- 参照先・出典を明記
- 用語の定義を最初に記載

## 最終チェックリスト

### 内容チェック

- [ ] 誤字脱字がないか
- [ ] 章立ての順番が正しいか（ストーリーが通るか）
- [ ] ページ番号を逐一確認
- [ ] ページ数の表記は合っているか
- [ ] グラフや表の数値の合計と部分の数は合っているか
- [ ] リード文だけ読んでストーリーが通るか

### デザインチェック

- [ ] ガイドからはみ出していないか
- [ ] 同じ図を流用している時は、図形のサイズなどが同じになっているか
- [ ] 一つのセルが分割されて印刷されていないか
- [ ] 箇条書きに余計な行間・インデントが含まれていないか
- [ ] フォントサイズは12pt以上か

### 最終確認

- [ ] スライドショーで通して確認
- [ ] 印刷プレビューで確認
- [ ] 第三者に5分で説明できるか
