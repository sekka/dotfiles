# タスク分解と実行管理

## 概要

タスク分解と体系的実行管理、進捗トラッキングを包括的に支援するスキルです。

## 実行フロー

### Step 1: タスク分解の準備

#### 計画ファイルの分析

- 全体の目標と範囲を把握
- 主要なマイルストーンを識別
- 技術的な依存関係を特定
- 成功基準の確認

#### 分解の基準

1. **1コミットの粒度**: 単一コミットで完結
2. **コンテキスト維持**: 単体で理解可能
3. **依存関係の明確化**: 実行順序を明示

### Step 2: 論理的な分割点の特定

#### 分割の観点

**機能的な境界:**

- 1つの機能 = 1つのタスク
- 独立してテスト可能な単位

**レイヤー境界:**

```
データベース層 → API層 → ビジネスロジック層 → UI層
```

**テスト可能な単位:**

- 単体テスト、統合テスト、E2Eテスト

### Step 3: コンテキスト注入

#### タスクに含めるべき情報

- **背景情報**: なぜ必要か、全体のどこに位置するか
- **参照情報**: 関連ドキュメント、参考コード
- **判断指針**: 迷いそうな点への指針

### Step 4: タスク出力の作成

#### タスクフォーマット

```markdown
## タスク [番号]: [タスク名]

### 概要
[達成すること]

### 背景コンテキスト
[必要な前提知識]

### 実装内容
- [ ] 具体的な作業項目

### 変更対象ファイル（予想）
- `path/to/file.ts`

### コミット情報
- プレフィックス: feat | fix | chore | docs
- メッセージ例: "feat: [変更内容]"

### 依存関係
- 前提タスク: [番号]
- 後続タスク: [番号]

### 完了条件
- [具体的な条件]

### 推定時間
- 30分 / 1時間 / 2時間
```

### Step 5: タスク品質の検証

#### 品質チェックリスト

- [ ] 各タスクは1コミットで完結するか
- [ ] 各タスクを単体で読んで実行可能か
- [ ] タスク間の依存関係は明確か
- [ ] 全タスク完了で元の計画が達成されるか
- [ ] タスクの粒度は適切か（30分〜2時間）

## タスク実行プロトコル

### 実行前の準備

1. タスクの要件を完全に理解
2. 必要なリソースとファイルを確認
3. 依存関係や前提条件を確認

### 実行中の管理

- 各ステップ完了時にタスクファイルを更新
- チェックボックスで完了ステータスを管理
- 問題発生時は即座に記録

### エラーハンドリング

1. エラー発生時: タスクファイルに詳細を記録
2. リカバリ可能: 対応策を記録してから実行
3. リカバリ不可能: 即座にユーザーに報告

## 出力成果物

1. **タスク分解結果**: マイルストーン、タスク一覧
2. **依存関係図**: 実行順序の可視化
3. **並行実行可能グループ**: 効率化のためのグルーピング
4. **完了条件**: フェーズ別、全体

## ベストプラクティス

1. **適切な粒度**: 1タスク = 30分〜2時間
2. **コンテキストを充実**: 単体で理解可能に
3. **依存関係を明確に**: ブロッカーを特定
4. **進捗を可視化**: ステータスを更新
5. **学びを記録**: 決定事項と理由
