# ISO/IEC 25010 品質特性チェックリスト

8つの品質特性とウェブ制作における具体的なチェック項目

---

## 概要

ISO/IEC 25010は、ソフトウェア品質モデルの国際規格です。
8つの品質特性と31の副特性により、非機能要件の漏れを防ぎます。

**重要:** 全ての品質特性を満たす必要はありません。プロジェクトの優先度に応じて選択してください。

---

## 品質特性の優先度付け

### 優先度の決め方

#### ステップ1: プロジェクトタイプを特定

| プロジェクトタイプ | 重視すべき品質特性（上位3つ） |
|-------------------|------------------------------|
| **コーポレートサイト** | 使用性、性能効率性、セキュリティ |
| **ECサイト** | セキュリティ、性能効率性、信頼性 |
| **SaaS** | 信頼性、セキュリティ、保守性 |
| **LP（ランディングページ）** | 性能効率性、使用性、互換性 |
| **管理画面** | 使用性、保守性、セキュリティ |

#### ステップ2: ステークホルダの期待を確認

```markdown
質問: "このプロジェクトで最も重要なことは何ですか？"

回答例:
- "速く表示されること" → 性能効率性（高）
- "誰でも使えること" → 使用性（高）
- "絶対に止まらないこと" → 信頼性（高）
- "個人情報を守ること" → セキュリティ（高）
```

#### ステップ3: 優先度を3段階で設定

- **高**: 必須、これがないとプロジェクト失敗
- **中**: 重要、あると良い
- **低**: 今回は無視（将来的に対応）

## 8つの品質特性

### 1. 機能適合性（Functional Suitability）

#### 定義

**必要な機能を提供しているか**

#### 副特性

1. **機能完全性**: 必要な機能が全て実装されているか
2. **機能正確性**: 機能が正しく動作するか
3. **機能適切性**: 目的に合った機能を提供しているか

#### ウェブ制作観点のチェックリスト

```markdown
## 機能適合性チェック

### 機能完全性
- [ ] 要件定義書の全機能が実装されているか
- [ ] ユーザーストーリーが全て満たされているか
- [ ] エッジケース（例外処理）が考慮されているか

### 機能正確性
- [ ] フォームバリデーションが正しく動作するか
- [ ] 検索結果が正確か
- [ ] 決済処理が正しく完了するか

### 機能適切性
- [ ] ユーザーのタスクを効率的に完了できるか
- [ ] 余計な機能がないか（Over-engineering防止）
- [ ] 想定外の使い方でもエラーが出ないか
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| 機能カバレッジ | 要件定義書との照合 | 100% |
| テストカバレッジ | Jest / Cypress | 80%以上 |
| エラー発生率 | Sentry / Google Analytics | 1%以下 |
```

---

### 2. 性能効率性（Performance Efficiency）

#### 定義

**速度、メモリ、リソース効率**

#### 副特性

1. **時間効率性**: レスポンス時間、処理時間
2. **リソース効率性**: メモリ、CPU、ネットワーク使用量
3. **容量**: 同時接続数、データ量の上限

#### ウェブ制作観点のチェックリスト

```markdown
## 性能効率性チェック

### 時間効率性（Core Web Vitals）
- [ ] LCP（Largest Contentful Paint）2.5秒以内
- [ ] FID（First Input Delay）100ms以内
- [ ] CLS（Cumulative Layout Shift）0.1以下
- [ ] TTFB（Time to First Byte）600ms以内

### リソース効率性
- [ ] 画像が最適化されているか（WebP/AVIF）
- [ ] バンドルサイズが適切か（初回ロード200KB以下）
- [ ] 不要なJavaScriptが読み込まれていないか
- [ ] lazy-load（遅延読み込み）が実装されているか

### 容量
- [ ] 同時接続数の想定は？（例: 100人/分）
- [ ] データ量の増加に対応できるか（例: 1万件の製品）
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| LCP | Lighthouse | 2.5秒以内 |
| バンドルサイズ | Webpack Bundle Analyzer | 200KB以下 |
| 同時接続 | Load testing (k6) | 100人/分で応答時間3秒以内 |
```

#### 優先度の決め方

```markdown
高優先度:
- LP、ECサイト（直帰率に直結）
- SEOが重要なサイト

中優先度:
- コーポレートサイト
- 管理画面（社内ツール）

低優先度:
- 極少人数が使う内部ツール
```

### 3. 互換性（Compatibility）

#### 定義

**ブラウザ、デバイス、既存システムとの統合**

#### 副特性

1. **共存性**: 他のシステムと共存できるか
2. **相互運用性**: 他のシステムと連携できるか

#### ウェブ制作観点のチェックリスト

```markdown
## 互換性チェック

### ブラウザ互換性
- [ ] 対応ブラウザを定義したか
  - Chrome最新2バージョン
  - Safari最新2バージョン
  - Edge最新2バージョン
  - Firefox最新2バージョン
- [ ] モバイルブラウザは対応するか
  - iOS Safari
  - Chrome for Android
- [ ] IE11対応が必要か（通常は不要）

### デバイス互換性
- [ ] 対応デバイスを定義したか
  - スマートフォン（375px〜）
  - タブレット（768px〜）
  - PC（1024px〜）
- [ ] タッチデバイス対応か
- [ ] 高解像度ディスプレイ対応か（Retina等）

### システム連携
- [ ] 既存システムとAPI連携するか
- [ ] データフォーマットは互換性があるか（JSON, XML）
- [ ] 認証方式は統一されているか
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| ブラウザ対応 | BrowserStack / 実機テスト | 対象ブラウザ全て |
| レスポンシブ | Chrome DevTools | 375px〜1920px全て |
| API互換性 | Postman / 統合テスト | エラー率0% |
```

### 4. 使用性（Usability）

#### 定義

**UI/UX、学習コスト、アクセシビリティ**

#### 副特性

1. **適切度認識性**: 何ができるか分かりやすいか
2. **習得性**: すぐに使い方を理解できるか
3. **運用性**: 操作しやすいか
4. **ユーザエラー防止性**: ミスを防げるか
5. **ユーザインタフェース快美性**: 見た目が美しいか
6. **アクセシビリティ**: 誰でも使えるか

#### ウェブ制作観点のチェックリスト

```markdown
## 使用性チェック

### 適切度認識性
- [ ] ファーストビューで何のサイトか分かるか
- [ ] CTAが明確か
- [ ] ナビゲーションが直感的か

### 習得性
- [ ] 初回訪問者がタスクを完了できるか
- [ ] オンボーディングが必要か
- [ ] ヘルプ/FAQが充実しているか

### 運用性
- [ ] フォーム入力が簡単か（項目数5つ以内）
- [ ] エラーメッセージが分かりやすいか
- [ ] 操作に対するフィードバックがあるか

### ユーザエラー防止性
- [ ] リアルタイムバリデーションがあるか
- [ ] 確認ダイアログがあるか（削除、送信前）
- [ ] 入力例（プレースホルダー）があるか

### ユーザインタフェース快美性
- [ ] デザインシステムが統一されているか
- [ ] 色のコントラストが適切か
- [ ] タイポグラフィが読みやすいか

### アクセシビリティ
- [ ] WCAG準拠レベルは？（A / AA / AAA）
- [ ] スクリーンリーダー対応か
- [ ] キーボード操作のみで使えるか
- [ ] alt属性が全画像に設定されているか
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| タスク完了率 | ユーザビリティテスト | 80%以上 |
| フォーム完了率 | Google Analytics | 70%以上 |
| WCAG準拠 | axe DevTools / WAVE | AA準拠 |
| キーボード操作 | 手動テスト | 全機能アクセス可能 |
```

### 5. 信頼性（Reliability）

#### 定義

**可用性、障害許容性、回復性**

#### 副特性

1. **成熟性**: バグが少ないか
2. **可用性**: システムが停止しないか
3. **障害許容性**: エラーが発生しても動作するか
4. **回復性**: 障害から復旧できるか

#### ウェブ制作観点のチェックリスト

```markdown
## 信頼性チェック

### 成熟性
- [ ] バグ管理システムがあるか（GitHub Issues, Jira）
- [ ] バグの優先度付けがされているか
- [ ] リグレッションテストがあるか

### 可用性
- [ ] 稼働率の目標は？（例: 99.9% = 月間43分まで停止）
- [ ] ホスティングサービスのSLAは？
  - Vercel: 99.99%
  - Netlify: 99.95%
  - AWS: 99.99%
- [ ] 監視ツールが導入されているか（UptimeRobot, Pingdom）

### 障害許容性
- [ ] ネットワークエラー時の挙動は？（オフライン対応）
- [ ] APIエラー時にフォールバックがあるか
- [ ] 外部サービス停止時の代替手段は？

### 回復性
- [ ] バックアップ戦略は？（日次、週次）
- [ ] 障害時の復旧手順書があるか
- [ ] RTO（目標復旧時間）は？（例: 1時間以内）
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| 稼働率 | Vercel Analytics / UptimeRobot | 99.9%以上 |
| MTTR（平均復旧時間） | インシデント記録 | 1時間以内 |
| バグ発生率 | GitHub Issues | 月間10件以下 |
```

### 6. セキュリティ（Security）

#### 定義

**認証、暗号化、データ保護**

#### 副特性

1. **機密性**: データが漏洩しないか
2. **インテグリティ**: データが改ざんされないか
3. **否認防止性**: 操作履歴が追跡できるか
4. **責任追跡性**: 誰が何をしたか記録されるか
5. **真正性**: 正当なユーザーか検証できるか

#### ウェブ制作観点のチェックリスト

```markdown
## セキュリティチェック

### 機密性
- [ ] HTTPSが全ページで有効か
- [ ] 個人情報を扱うか（氏名、メール、電話番号）
- [ ] 個人情報は暗号化されているか
- [ ] 環境変数（.env）がGitにコミットされていないか

### インテグリティ
- [ ] CSRF対策があるか
- [ ] SQLインジェクション対策があるか
- [ ] XSS対策があるか
- [ ] 入力値のサニタイズがされているか

### 否認防止性
- [ ] ログイン履歴が記録されるか
- [ ] 重要な操作（削除、変更）がログに残るか

### 責任追跡性
- [ ] 誰が何をしたか追跡できるか
- [ ] 監査ログがあるか

### 真正性
- [ ] 認証方式は？（パスワード、JWT、OAuth）
- [ ] 二要素認証（2FA）が必要か
- [ ] セッション管理が適切か
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| HTTPS | SSL Labs | A+評価 |
| 脆弱性スキャン | npm audit / Snyk | 重大な脆弱性0件 |
| OWASP Top 10 | 手動監査 / OWASP ZAP | 全項目対策済み |
```

#### OWASP Top 10チェックリスト

```markdown
- [ ] A01: アクセス制御の不備
- [ ] A02: 暗号化の失敗
- [ ] A03: インジェクション
- [ ] A04: 安全でない設計
- [ ] A05: セキュリティ設定のミス
- [ ] A06: 脆弱で古いコンポーネント
- [ ] A07: 識別と認証の失敗
- [ ] A08: ソフトウェアとデータの整合性の不備
- [ ] A09: セキュリティログとモニタリングの失敗
- [ ] A10: サーバーサイドリクエストフォージェリ（SSRF）
```

### 7. 保守性（Maintainability）

#### 定義

**コード品質、拡張性、テスト容易性**

#### 副特性

1. **モジュール性**: 機能が独立しているか
2. **再利用性**: コードを再利用できるか
3. **解析性**: コードが理解しやすいか
4. **修正性**: バグを修正しやすいか
5. **試験性**: テストしやすいか

#### ウェブ制作観点のチェックリスト

```markdown
## 保守性チェック

### モジュール性
- [ ] コンポーネントが適切に分割されているか
- [ ] 疎結合か（コンポーネント間の依存が少ない）
- [ ] 単一責任の原則を守っているか

### 再利用性
- [ ] 共通コンポーネントが抽出されているか
- [ ] UIライブラリ（Radix UI等）を使っているか
- [ ] カスタムフックが再利用可能か

### 解析性
- [ ] コードに適切なコメントがあるか
- [ ] 変数名・関数名が分かりやすいか
- [ ] READMEがあるか

### 修正性
- [ ] ESLintが導入されているか
- [ ] Prettierでフォーマット統一されているか
- [ ] TypeScriptで型安全か

### 試験性
- [ ] ユニットテストがあるか
- [ ] E2Eテストがあるか
- [ ] テストカバレッジ目標は？（例: 80%）
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| コード品質 | ESLint | エラー0件 |
| テストカバレッジ | Jest | 80%以上 |
| 循環的複雑度 | SonarQube | 10以下 |
| 技術的負債 | SonarQube | A評価 |
```

### 8. 移植性（Portability）

#### 定義

**環境依存、デプロイ容易性**

#### 副特性

1. **適応性**: 異なる環境で動作するか
2. **設置性**: 簡単にインストールできるか
3. **置換性**: 他のシステムに置き換えられるか

#### ウェブ制作観点のチェックリスト

```markdown
## 移植性チェック

### 適応性
- [ ] 開発環境と本番環境で動作が同じか
- [ ] 環境変数で設定を切り替えられるか
- [ ] ホスティングサービスに依存していないか

### 設置性
- [ ] デプロイが簡単か（ワンコマンド）
- [ ] CI/CDパイプラインがあるか
- [ ] ドキュメントがあるか（デプロイ手順書）

### 置換性
- [ ] ベンダーロックインを避けているか
- [ ] データエクスポートが可能か
- [ ] 別のホスティングサービスに移行可能か
```

#### 測定方法

```markdown
| 測定項目 | 測定方法 | 目標値 |
|----------|----------|--------|
| デプロイ時間 | CI/CD | 10分以内 |
| 環境差異 | テスト | 開発=本番 |
```

## 優先度の決定フローチャート

```
品質特性を1つ選ぶ
  ↓
Q1: これがないとプロジェクトが失敗する？
  YES → 優先度「高」
  NO  → 次へ
  ↓
Q2: ユーザー体験に大きく影響する？
  YES → 優先度「高」
  NO  → 次へ
  ↓
Q3: 法令・規制で必須？
  YES → 優先度「高」
  NO  → 次へ
  ↓
Q4: あると良いが、なくても問題ない？
  YES → 優先度「中」
  NO  → 優先度「低」
```

## 実践例

### シナリオ: ECサイト構築

#### Step 1: 8つの品質特性を全て列挙

```markdown
1. 機能適合性
2. 性能効率性
3. 互換性
4. 使用性
5. 信頼性
6. セキュリティ
7. 保守性
8. 移植性
```

#### Step 2: 優先度を付ける

```markdown
高: セキュリティ、性能効率性、信頼性
中: 使用性、互換性、保守性
低: 機能適合性、移植性
```

**理由:**
- セキュリティ: 決済情報を扱うため必須
- 性能効率性: ページ速度が売上に直結
- 信頼性: サイトが止まると売上損失
- 使用性: カート放棄率に影響
- 互換性: モバイル対応必須
- 保守性: 継続的に機能追加が必要
- 移植性: 今回は優先度低い

#### Step 3: 具体的な要件を定義

```markdown
| 品質特性 | 具体的な要件 | 測定方法 | 目標値 | 優先度 |
|----------|--------------|----------|--------|--------|
| セキュリティ | PCI DSS準拠 | Shopify標準 | 準拠 | 高 |
| | SSL証明書 | SSL Labs | A+評価 | 高 |
| 性能効率性 | LCP | Lighthouse | 2.5秒以内 | 高 |
| | 同時接続 | Load testing | 500人/分 | 高 |
| 信頼性 | 稼働率 | Vercel Analytics | 99.9% | 高 |
| | 決済成功率 | Shopify | 99.5% | 高 |
| 使用性 | カート放棄率 | Google Analytics | 30%以下 | 中 |
| | WCAG準拠 | axe DevTools | AA | 中 |
| 互換性 | モバイル対応 | 実機テスト | iOS/Android | 中 |
```
