# デザインシステムのコンポーネントガイドライン整備

> 出典: https://note.com/yasuun/n/nfeee40ddb751
> 著者: yasuka（10X プロダクトデザイナー）
> 追加日: 2026-04-13

デザインシステムのコンポーネントガイドラインをゼロから整備した実践事例。
Figma + Storybook を組み合わせた運用設計と、課題解決のアプローチ。

## 運用体制の前提

- **Figma**: デザイントークンとコンポーネントを管理
- **Storybook**: Web アプリ向けコンポーネントカタログを運用
- **複数デザインシステム**: プロダクトごとに異なるシステムを並行運用

## 主な課題

- 「見た目が似ているコンポーネントが複数あり、用途も曖昧」
- コンポーネントの使い分けルールが整備されていない
- 開発者・デザイナー間で使うコンポーネントの選択がばらつく

## ガイドライン整備の目的

1. コンポーネントの**役割・用途の明確化**
2. プロダクト横断での**一貫性向上**
3. 似たコンポーネントの**使い分けルール**の定義
4. 開発効率の向上

## ガイドラインに含めるべき内容

- コンポーネントの**使用目的**（何のために使うか）
- **使用場面**（どこで使うか）
- **使い分け基準**（似たコンポーネントとの違い）
- **DO/DON'T**（良い例・悪い例）
- **アクセシビリティ考慮点**

## Figma + Storybook 連携のポイント

```
Figma（デザイン側）
└── コンポーネント定義 + デザイントークン
    └── ガイドライン文書（説明・DO/DON'T）

Storybook（実装側）
└── コンポーネントカタログ
    └── Controls でバリアント確認
    └── Docs タブにガイドライン記載
```

## ユースケース

- 複数人でデザインシステムを運用するとき
- 似たコンポーネントが乱立して混乱が生じているとき
- デザイナーと開発者の間でコンポーネント選択がばらつくとき
