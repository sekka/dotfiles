# カラースケール設計 - OKLCHで壊れないパレットを作る

## 概要

デザインシステムにおけるカラースケール（色番手 100-900）を、OKLCHの知覚的均一性を活用して構築する方法論。自動生成 → 目視診断 → 手動補正の5フェーズで、トーンの揃ったパレットを作る。

> Added: 2026-04-08

## なぜトーンを揃える必要があるか

### 問題：同じ番手でもトーンがバラつく

ステータスバッジ（Success/Error/Warning/Info）を同じ600番で並べた場合：

- 赤だけ文字が読みにくい
- 黄だけ目立つ
- 青だけ沈んで見える

HSL では数値が同じでも知覚的明度が異なるため、色相を変えると破綻する。OKLCH なら同じ Lightness 値で知覚的にもトーンが統一される。

### 効果

- 色が変わっても UI の品質が一定に保たれる
- コントラスト比の管理が容易
- デザイントークンの拡張性が向上

---

## 5フェーズプロセス

### Phase 1: Draft（ドラフト）

80点の素材を自動生成する。ゼロから作らない。

- **Figma プラグイン PencilColor** でベースパレットを生成
- **Tailwind CSS のカラーパレット** を出発点にする
- ブランドカラーのみ手動で作成

### Phase 2: Diagnosis（診断）

生成した色をグレースケール表示で検証する。**このステップを省くと展開時に必ず破綻する。**

- Figma でレイヤーをグレースケール表示に切り替え
- 同じ番手の色が同じグレーに見えるか確認
- 数値は正しくても視覚的違和感があれば問題あり

### Phase 3: Tuning（調整）

グレースケール診断で見つけた違和感を手動補正する。

- **「数値よりも見た目」を信じる**
- 知覚的な一貫性を最優先
- Huetone でコントラスト比を確認しながら調整

### Phase 4: Fix（確定）

調整済みスケールから UI の基準色を決定する。

- 実装用の色番を確定（通常は 600 番が基準色）
- デザイントークンとして命名

### Phase 5: Alpha（透過色）

素材として透過色を定義する。

- **Black α**（黒の透過色）: オーバーレイ、シャドウ用
- **White α**（白の透過色）: ハイライト、ホバー状態用

## 黄色の例外ルール

### 問題の本質

黄色は他の色相と「光の性質が根本的に異なる」。知覚輝度が本来高いため、グレースケール上の整合性を追求すると**濁った茶色に変質**し、実用性が著しく低下する。

### 解決策：黄色のみルール変更

1. **グレースケール整合性を一旦無視**する
2. **色付き状態での見た目**でバランス調整
3. UI 基準色は **600番ではなく400番** を採用
4. 文字色は **白ではなく黒** を使用

### 適用範囲

この特例ルールは黄色だけでなく、**知覚輝度が高い色相全般**に適用可能：

- 蛍光グリーン
- ピンク
- ライム

### 設計原則：「例外をルール化する」

例外を放置せず、明示的なルールとしてシステムに組み込む。これはカラーシステムに限らず、デザインシステム全体に通じる原則。

## ユースケース

- **デザインシステムのカラートークン構築**: 新規デザインシステム立ち上げ時
- **既存パレットの品質監査**: グレースケール診断で現状の問題を発見
- **ステータスカラーの統一**: Success/Error/Warning/Info 等のセマンティックカラー
- **ダークモード対応**: トーンが揃っていればライト↔ダーク変換が容易

---

## 注意点

- Phase 2（診断）を省略すると、スケール全体が破綻するリスクが高い
- 「数値的正しさ」と「視覚的正しさ」は異なる — 最終判断は必ず目視で行う
- 次回記事予告：「3階層モデルと State Layer で『トークンの増殖』を止める」
