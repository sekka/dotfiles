# タイプスケールの決め方（設計判断フレームワーク）

> 出典: https://note.com/hirotoarakawa/n/n6924a725d584
> 執筆日: 2022-12-06
> 追加日: 2026-04-13

UI/UX デザイナー視点でのタイプスケール（フォントサイズのルール）設計フレームワーク。
CSS 実装よりも「どの比率ルールをどの用途に選ぶか」という設計判断に焦点を当てている。

> 関連: CSS 実装は [typography-scaling.md](./typography-scaling.md)、流体タイポグラフィは [fluid-type-scale.md](./fluid-type-scale.md)

## タイプスケールの 2 つの比率ルール

### 1. 乗算のルール（Multiplicative Scale）

一定の倍率でフォントサイズが大きくなる。音楽の音階や黄金比を基にした比率が多い。

**例: Major Third（× 1.25）**
```
10, 13, 16, 20, 25, 31, 39px...
```

**特徴:**
- 抑揚のあるレイアウトが作れる
- 標準サイズ（16px）付近に細かいバリエーションが多い → 実用的
- 行間やマージンにも同じ比率を適用すると、リズミカルなレイアウトになる

**ツール:** [Modular Scale Generator](https://www.modularscale.com/)

### 2. 加算のルール（Additive Scale）

一定数ずつ加算（減算）される。日本の「号」活字が代表例。

**例: 加算 +2px（16px 基準）**
```
10, 12, 14, 16, 18, 20, 22px...
```

**特徴:**
- ノートの罫線に沿うようなグリッドレイアウトが作りやすい
- 一直線に変化するため、単調になりやすい
- 大きいサイズに選択肢が多すぎ、どれにするか悩みやすい

## STEP 1：スケールの比率の「大きさ」を決める

サービスの用途によって比率の大きさを選ぶ。

| 用途 | 推奨比率 | 理由 |
|------|---------|------|
| マーケティングサイト、LP、ブランドサイト | **大きい比率** (1.333 〜 1.5 など) | タイトル・見出しのメッセージを強調できる |
| EC サイト、情報量の多いサイト | **小さい比率** (1.125 〜 1.2 など) | 一覧性が高まり、より多くの情報を並べられる |
| スマートフォン向けアプリ | **小さい比率** | 小画面では大きすぎるフォントが配置しにくい |

## STEP 2：比率の「ルール」を決める

```
グリッドに沿ったレイアウトが必要か？
  ├─ Yes → 加算のルール
  └─ No  → 乗算のルール（基本的にこちらを選べばOK）
```

**乗算のルールを基本とする理由:**
- 16px 付近（標準サイズ）に実用的な細かいバリエーションが生まれる
- 加算のルールは大きいサイズに選択肢が多すぎて選定が難しくなる
- Figma・ノーコードツールなど、複雑なグリッドレイアウトのアプリでは加算が有効

## デザインシステムとしての定義

プロダクト（アプリ）とマーケティングサイトで同じフォントサイズを共有する場合:

- **推奨: Major Second（× 1.125）などの小さい比率 + 乗算のルール**
- 小さい比率でも大きいフォントサイズを複数用意できる
- マーケティングサイト側は、用意されたフォントサイズの中から選べるようにする

> 大切なのは「タイプスケールを定義すること」そのもの。どのルールでも、統一されていれば一貫性は保てる。

## Figma プラグイン

[Typescale.io](https://typescale.io/) — 各種タイプスケールを選択して生成できる。小数点の丸めも可能。

## ユースケース

- 新規プロダクトのタイプスケールを設計するとき
- デザインシステムのフォントサイズトークンを定義するとき
- サービスの方向性（マーケ vs アプリ）に合わせた比率を選ぶとき

## 注意点

- 2022年の記事のため、数値の具体例は参考程度に
- 日本語フォントは欧文より字間が詰まりやすいため、比率の感覚が異なる場合がある
  （→ ハーモニックスケールが日本語に向いている理由は [typography-scaling.md](./typography-scaling.md) を参照）
