# OKLCHで「壊れないカラースケール」を作る

著者: Sho Tsusaka（フリーランスのアートディレクター/UIUXデザイナー）
公開日: 2026年4月4日
トピック: Figma、デザイン、UIデザイン、デザインシステム、カラーシステム

## 主要なポイント

### 1. OKLCHが必要な理由

従来のカラーモデルの問題点：
Hex、RGB、HSLは「数値の変化と人間の目の感じ方が一致しない」。例えば、HSLで明度を50%に統一して色相を変えると、黄色は眩しく見え、青は暗く見える。これにより、同じ番手（例：500番）の色を揃えようとしてもトーンがバラバラになる。

OKLCHの利点：
「知覚的均一性」を持つため、数値変化が人間の知覚と一致し、色相が異なっても一貫したトーンを保つ。アクセシビリティ（コントラスト比）管理も容易。

## 5フェーズの素材製作プロセス

### Phase 1: Draft（ドラフト）
- FigmaプラグインのPencilColorやTailwind CSSのパレットをベースに80点の素材を自動生成
- ブランドカラーのみ手動で作成
- ゼロからの製作は避ける

### Phase 2: Diagnosis（診断）
- 生成した色をグレースケール表示で確認
- 数値は正しくても視覚的違和感を検出
- この診断ステップを省くと展開時に必ず破綻する

### Phase 3: Tuning（調整）
- グレースケール診断で見つけた違和感を手動補正
- 「数値よりも見た目」を信じる
- 知覚的な一貫性を優先

### Phase 4: Fix（確定）
- 調整済みスケールからUIの基準色を決定
- 実装用の色番を確定

### Phase 5: Alpha（透過色）
- Black α（黒の透過色）
- White α（白の透過色）
- を素材として定義

## トーンを揃える重要性

実例：ステータスバッジ
Success（緑）、Error（赤）、Warning（黄）、Info（青）の4色を同じ番手（600番など）で並べた場合、トーンが揃っていなければ：
- 赤だけ文字が読みにくい
- 黄だけ目立つ
- 青だけ沈んで見える

トーンの統一により、色が変わってもUIの品質が一定に保たれる。

## Yellow（黄色）の特例ルール

### 問題の本質
黄色は他の色相と「光の性質が根本的に異なる」。知覚輝度が本来高いため、グレースケール上の整合性を追求すると濁った茶色に変質し、実用性が著しく低下。

### 解決策：黄色のみルール変更
1. グレースケール整合性を一旦無視
2. 色付き状態での見た目でバランス調整
3. UI基準色は600番ではなく400番採用
4. 文字色は白ではなく黒を使用

この特例ルールは、蛍光グリーンやピンクなど知覚輝度が高い色相全般に適用可能。

### 核となる考え方
「例外をルール化する」——システム設計における重要な原則。

## まとめ

1. Hex/RGB/HSLは数値と見た目が不一致でシステム設計に不向き
2. OKLCHの知覚的均一性により、異なる色相でもトーン整合性が保証される
3. 5フェーズプロセスにより、自動生成後の目視診断・補正が必須
4. 黄色は例外ルール（400番+黒文字）で運用
