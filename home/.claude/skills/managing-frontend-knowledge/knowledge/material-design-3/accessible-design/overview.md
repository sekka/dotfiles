# Accessible Design Overview（アクセシブルデザイン概要）

Material Design 3は、アクセシビリティファーストの設計を採用しています。すべてのユーザーが、能力や状況に関わらず、快適に使えるUIを目指します。

## アクセシビリティの重要性

### なぜアクセシビリティが重要か

1. **インクルージョン**: すべての人が情報とサービスにアクセスできる権利
2. **ビジネス価値**: 世界人口の15%（10億人以上）が何らかの障害を持つ
3. **法的要件**: 多くの国でアクセシビリティが法的に義務付けられている
4. **UX向上**: アクセシブルなデザインはすべてのユーザーにとって使いやすい

### Universal Design（ユニバーサルデザイン）

Material Design 3は、特定のユーザーだけでなく、すべてのユーザーに利益をもたらすデザインを目指します：

- **明確なコントラスト**: 視覚障害のあるユーザーだけでなく、明るい屋外でも見やすい
- **大きなタップ領域**: 運動障害のあるユーザーだけでなく、すべてのユーザーが正確にタップしやすい
- **キーボードナビゲーション**: スクリーンリーダーユーザーだけでなく、パワーユーザーも高速操作可能

## 4つのアクセシビリティの柱

### 1. 知覚可能（Perceivable）

**原則**: 情報とUIコンポーネントは、ユーザーが知覚できる方法で提示されなければならない。

**実践**:
- **色覚異常**: 色だけに依存しない情報伝達
- **コントラスト**: WCAG AA以上のコントラスト比
- **代替テキスト**: 画像に意味のあるaltテキスト
- **テキストサイズ**: 拡大可能なテキスト（最大200%）

### 2. 操作可能（Operable）

**原則**: UIコンポーネントとナビゲーションは操作可能でなければならない。

**実践**:
- **キーボードアクセス**: すべての機能がキーボードで操作可能
- **十分な時間**: タイムアウトを避けるか、延長オプションを提供
- **発作の防止**: 点滅や閃光を避ける（1秒間に3回以上）
- **ナビゲーション**: 明確で一貫したナビゲーション構造

### 3. 理解可能（Understandable）

**原則**: 情報とUIの操作方法は理解可能でなければならない。

**実践**:
- **明確な言語**: 簡潔で理解しやすいテキスト
- **予測可能**: 一貫した動作とレイアウト
- **エラー防止**: エラーを防ぎ、発生時には明確な修正方法を提示
- **ヘルプとガイダンス**: コンテキストに応じたヘルプ

### 4. 堅牢（Robust）

**原則**: コンテンツは、支援技術を含む様々なユーザーエージェントで確実に解釈できなければならない。

**実践**:
- **セマンティックHTML**: 意味のあるHTML要素の使用
- **ARIA属性**: 必要に応じてARIA属性で補完
- **互換性**: スクリーンリーダー、音声認識などに対応
- **標準準拠**: Web標準とガイドラインに従う

## WCAG 2.1との関係

Material Design 3は、WCAG（Web Content Accessibility Guidelines）2.1のレベルAA準拠を目指しています。

### WCAGのレベル

- **レベルA**: 最低限のアクセシビリティ
- **レベルAA**: 推奨される標準（Material Design 3の目標）
- **レベルAAA**: 最高レベル（すべてのケースでは実現困難）

### 主要な WCAG 基準

1. **1.4.3 コントラスト（最低限）**: 4.5:1（通常テキスト）、3:1（大きなテキスト）
2. **1.4.11 非テキストコントラスト**: UI要素は3:1以上
3. **2.1.1 キーボード**: すべての機能がキーボードアクセス可能
4. **2.4.7 フォーカスの可視化**: キーボードフォーカスが明確に見える
5. **4.1.2 名前、役割、値**: すべてのUI要素が支援技術に適切に認識される

## 障害の種類と対応

### 視覚障害

**種類**:
- 全盲
- 低視力
- 色覚異常（色盲）

**対応**:
- スクリーンリーダー対応（適切なHTML、ARIA）
- 高コントラスト
- テキストの拡大可能性
- 色以外の手がかり（アイコン、テキストラベル）

### 聴覚障害

**種類**:
- 全聾
- 難聴

**対応**:
- 字幕・キャプション
- 視覚的なアラート（音だけに依存しない）
- テキストベースの代替手段

### 運動障害

**種類**:
- 手の震え
- 麻痺
- 限定的な細かい運動制御

**対応**:
- 大きなタップ領域（48x48dp以上）
- キーボードショートカット
- 音声コントロール
- ドラッグ＆ドロップの代替手段

### 認知障害

**種類**:
- 学習障害
- 注意欠陥
- 記憶障害

**対応**:
- シンプルで一貫したUI
- 明確な指示とフィードバック
- エラー防止とリカバリー
- 段階的な情報開示

## Material Design 3の組み込みアクセシビリティ

### 自動的に提供される機能

1. **カラーシステム**: WCAG AA準拠のコントラスト
2. **タイポグラフィ**: 読みやすいフォントサイズと行間
3. **タップ領域**: 48x48dp以上の推奨サイズ
4. **フォーカスインディケータ**: 明確なキーボードフォーカス
5. **セマンティックコンポーネント**: 適切なHTML要素とARIA

### デザイナー・開発者の責任

Material Design 3を使用しても、以下は手動で対応が必要：

1. **代替テキスト**: 画像とアイコンに意味のあるaltテキスト
2. **フォームラベル**: すべての入力フィールドに明確なラベル
3. **エラーメッセージ**: 具体的で理解しやすいエラー説明
4. **キーボードナビゲーション**: 論理的なTab順序
5. **コンテンツの構造**: 適切な見出し階層

## アクセシビリティテスト

### 手動テスト

1. **キーボードのみで操作**: マウスを使わず、すべての機能にアクセスできるか
2. **スクリーンリーダー**: NVDA（Windows）、JAWS、VoiceOver（Mac/iOS）、TalkBack（Android）でテスト
3. **ズーム**: 200%にズームしても使えるか
4. **色覚シミュレーション**: 色覚異常シミュレーターで確認

### 自動テスト

1. **axe DevTools**: ブラウザ拡張でアクセシビリティ問題を検出
2. **Lighthouse**: Chromeの監査ツール
3. **WAVE**: Web Accessibility Evaluation Tool

### ユーザーテスト

- 実際の障害を持つユーザーでテスト
- 支援技術を使用したユーザビリティテスト

## ベストプラクティス

### DO（推奨）

1. **最初から組み込む**: 後付けではなく、設計段階から考慮
2. **実際のユーザーでテスト**: 障害を持つユーザーを含める
3. **継続的な改善**: アクセシビリティは一度きりではなく継続的な取り組み
4. **教育**: チーム全体がアクセシビリティの重要性を理解

### DON'T（避けるべき）

1. **後回しにしない**: リリース直前の対応は困難
2. **自動テストだけに依存**: 手動テストとユーザーテストも必須
3. **例外を作らない**: すべての機能がアクセシブルであるべき
4. **最低限で満足しない**: WCAG AAは最低ライン、さらなる改善を目指す
