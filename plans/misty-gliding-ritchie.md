# インタビューファースト原則の導入

## Context

Zenn記事「AIに指示するのではなく、まずはAIに質問させた方が捗る」の手法を取り入れたい。
記事の核心: タスク着手前にAIがユーザーにインタビューし、暗黙知を引き出すことで回答精度を上げる。

**3原則:**

1. インタビュー目的を明示
2. 表面的・自明な質問を避け、最大3問に制限
3. 途中で切り上げられる終了条件

**現状:** CLAUDE.mdに「曖昧な要件はAskUserQuestionで必ず確認」の1行があるのみ。既存の `/interview` コマンドはプランファイル専用で用途が異なる。

## 方針: `rules/interview-first.md` を1ファイル追加

**選定理由:**

- `~/.claude/rules/` はClaude Codeが自動読み込み（現セッションで確認済み）
- 1ファイル追加・削除で完結、既存ファイル変更なし → 最も剥がしやすい
- 既存ルール (`plan-review.md` 4行, `security.md` 7行) と同じ粒度

**不採用の選択肢:**

- CLAUDE.md強化 → diffで剥がす手間、関心事の混在
- `/interview` 汎用化 → 明示起動が必要で記事の趣旨とズレる
- 新スキル → YAGNI違反

## 実装

### Step 1: ルールファイル作成

**ファイル:** `home/.claude/rules/interview-first.md`

```markdown
# インタビューファースト原則

タスク開始時、すぐに実装に取りかからず、まずユーザーにインタビューして要件を明確化する。

## いつ適用するか

- 新機能の設計・実装を依頼されたとき
- 曖昧さや複数の解釈が残る要件を受けたとき
- 要件が1行の指示や「〜してほしい」のような概要レベルのとき

## いつ適用しないか

- 具体的な手順が明示されているとき（「このファイルのN行目をXに変更して」等）
- バグ修正で再現手順が明確なとき
- 既に十分なコンテキストがあるとき

## インタビューの進め方

1. **目的を明示してから開始**: 「〜を正確に理解するために質問させてください」と宣言
2. **表面的・自明な質問を避け、最大3問に絞る**: コードや文脈から読み取れることは自分で調べる。ユーザーにしか答えられない判断・意図・優先度を聞く
3. **終了条件**: 十分な情報が得られたら切り上げて次に進む。「他に確認したいことはありますか？」で終了の機会を与える
```

### Step 2: 動作確認

新しいClaude Codeセッションで曖昧な依頼（例: 「認証機能を追加して」）を投げ、インタビューから始まるか確認。

## 検証

- [ ] `home/.claude/rules/interview-first.md` がシンボリックリンク経由で `~/.claude/rules/` に配置される
- [ ] 新セッションのシステムプロンプトにルール内容が含まれる
- [ ] 曖昧な依頼時 → インタビューが走る
- [ ] 明確な依頼時 → インタビューをスキップして作業開始

## 剥がし方

```bash
git rm home/.claude/rules/interview-first.md
```
